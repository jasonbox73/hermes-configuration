Agentic Hermes MVP — Work System

Four agents. One Kanban board as the source of truth. Every request becomes a ticket and moves from vague intent to a reviewed artifact — research, documents, software, troubleshooting, planning, automation.

overview

Every user request becomes a Kanban ticket that travels a generic lifecycle. The board is the source of truth — agent memory is not. Each ticket gets its own isolated workspace so unrelated work never cross-contaminates.

Capture → Specify → Research → Build → Review → Revise → Done → Archive
NOTE — the setup script does not execute any hermes command. It scaffolds local files and prints every Hermes command for you to run by hand.

the agents

Each profile has its own identity file at ~/.hermes/profiles/<name>/SOUL.md, staged for you under profiles/.

routing flows

The Chief picks a path based on ticket type. Three common shapes:

// assignment paths

Mechanism: dependency graph. Chief (orchestrator) creates one task per role and links them with --parent; the dispatcher auto-promotes a child todo→ready when its parents are done. Each worker completes its own card with kanban_complete; only chief/operator finalizes and archives.

dispatch model

The gateway hosts a dispatcher that runs on an interval (default 60s). It atomically claims ready tickets, spawns the assigned profile as a worker with HERMES_KANBAN_TASK pinned, and the worker drives the board through Kanban tools — never the CLI.

hermes commands

Printed by the setup script; run them yourself. Verified against the Hermes docs.

1 · create the four profiles

hermes profile create chief --clone-from default --description "Orchestrates Kanban work, clarifies requests, creates tickets…"
hermes profile create researcher --clone-from default --description "Researches, analyzes, compares options, diagnoses problems…"
hermes profile create builder --clone-from default --description "Creates requested artifacts: code, scripts, documents, plans…"
hermes profile create reviewer --clone-from default --description "Reviews completed work against acceptance criteria…"

2 · confirm paths, then place each SOUL.md

hermes profile show chief        # repeat for researcher / builder / reviewer
cp profiles/chief/SOUL.md      ~/.hermes/profiles/chief/SOUL.md
cp profiles/researcher/SOUL.md ~/.hermes/profiles/researcher/SOUL.md
cp profiles/builder/SOUL.md    ~/.hermes/profiles/builder/SOUL.md
cp profiles/reviewer/SOUL.md   ~/.hermes/profiles/reviewer/SOUL.md

3 · point each profile at the workspace

chief      config set terminal.cwd ~/HermesWork/agentic-mvp
researcher config set terminal.cwd ~/HermesWork/agentic-mvp
builder    config set terminal.cwd ~/HermesWork/agentic-mvp
reviewer   config set terminal.cwd ~/HermesWork/agentic-mvp

4 · configure Chief as orchestrator + WIP limits

chief config set kanban.orchestrator_profile chief
chief config set kanban.default_assignee chief
chief config set kanban.auto_decompose false   # explicit graph (may already be false)
chief config set kanban.max_in_progress 3      # PRD WIP target

# edit Chief config; both lists must contain:
toolsets: [hermes-cli, kanban]
platform_toolsets:
  cli: [hermes-cli, kanban]

chief skills list | grep kanban-orchestrator || chief skills reset kanban-orchestrator --restore --yes
researcher skills list | grep kanban-worker || researcher skills reset kanban-worker --restore --yes
builder skills list | grep kanban-worker || builder skills reset kanban-worker --restore --yes
reviewer skills list | grep kanban-worker || reviewer skills reset kanban-worker --restore --yes

5 · initialize Kanban & create the board

hermes kanban init
hermes kanban boards create agentic-mvp \
  --name "Agentic Hermes MVP" \
  --description "Generic agentic work system using chief, researcher, builder, and reviewer profiles." \
  --switch
hermes kanban boards list
hermes kanban boards show

6 · start the dispatcher & build the dependency graph

chief gateway start   # chief hosts the gateway/dispatcher

# related cards share one ticket-isolated workspace.
# the dispatcher auto-promotes a child todo→ready when its parents are done.
TICKET_WS="$HOME/HermesWork/agentic-mvp/workspaces/TICKET-0001"
mkdir -p "$TICKET_WS"
BUILD=$(hermes kanban create "Build: Agentic Hermes MVP overview" --assignee builder \
  --body "Create outputs/hello-hermes.md; explain all roles and build -> review -> finalize; no placeholders." \
  --workspace "dir:$TICKET_WS" --json | jq -r .id)
REVIEW=$(hermes kanban create "Review: Agentic Hermes MVP overview" --assignee reviewer --parent "$BUILD" \
  --body "Verify acceptance criteria and record a verdict." --workspace "dir:$TICKET_WS" --json | jq -r .id)
FINALIZE=$(hermes kanban create "Finalize: Agentic Hermes MVP overview" --assignee chief --parent "$REVIEW" \
  --body "Close only after accepted review." --workspace "dir:$TICKET_WS" --json | jq -r .id)

caveats

Required skills — verify kanban-orchestrator for Chief and kanban-worker for each worker; restore missing bundled skills before dispatch.
Revision gate — review remains blocked while a linked revision runs. Chief unblocks the same review only after revision completion.
SOUL.md location — read from HERMES_HOME (~/.hermes by default), never from the working directory, so identity files are staged under profiles/.
SAFETY — the script writes no secrets and never overwrites existing files without --force (context/templates) or --overwrite-soul (SOUL.md).
